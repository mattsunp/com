# Webデザイン設計の原理

## 最初の3秒が全てを決める

訪問者がページを開いた瞬間から3秒以内に、脳は「ここにいる価値があるか」を判断する。この3秒間に発生することは情報の処理ではなく**感情の反応**だ。

凡庸なWebディレクターはこの3秒を「タイトルと説明文で何を伝えるか」の問題として扱う。天才的なWebディレクターは「この3秒間に訪問者の中に何の感情を生むか」の問題として扱う。

**問うべき問いは**：このページを開いた0.5秒後に、訪問者の脳に浮かぶ感情は何か。驚きか。静謐か。引力か。圧倒か。期待か。それともなにもないか。

「なにもない」が最悪の答えだ。最良の答えは「この世界の空気を吸った」という感覚だ。

## パターン選択の判断原理

Seep Logos は16のWebパターン（A〜P）を持つ。テンプレートとして使うのではなく、**それぞれが生む感情状態**を理解して選択する。

| パターン | 生む感情状態 | 選ぶべき局面 |
|---|---|---|
| A（ティザー） | 「何かが始まる」という緊張・期待 | 発表前。情報は最小限でよいとき |
| B（ワールドファースト） | 世界への好奇心・引力 | 世界観から語りたいとき |
| C（エディトリアル） | 静謐・知的没入 | テキストの重みが武器のとき |
| D（ベントーリビール） | 整理された全貌の把握感 | 多情報を同時に整理したいとき |
| E（スクロールストーリー） | 物語に引き込まれる感覚 | スクロールで時間を体験させたいとき |
| F（シネマティックLP） | 映画的高揚感 | 映像・トレーラーが主役のとき |
| G（キャラクター図鑑） | キャラクターへの親しみ・推しの感覚 | キャラクター主体のIPのとき |
| H（ダークイマーシブ） | 異世界への没入・不安の美学 | SF・機械・クールな世界観のとき |
| I（和風テクスチャ） | 日本的叙情・時代感 | 和風世界観・縦書きが合うとき |
| J（ハブ＆スポーク） | 全体像の把握・選択の自由感 | 多コンテンツへの入口として機能させたいとき |
| K（パステルポップ） | かわいさ・親しみ・ポップな高揚感 | ライト層・キャラクター推し系のIPのとき |
| L（ニュースフィード） | 最新情報への期待・コミュニティの存在感 | 運用フェーズ・放映中・ライブ型のとき |
| M（タイムラインイマーション） | 歴史の重さ・時間の流れへの没入 | 年表・歴史・時系列を語るとき |
| N（チャプタースクロール） | 章を読み進める充実感・重厚感 | 深い物語を読ませたいとき |
| O（カードカルーセル） | カテゴリ横断の発見感 | ポータル的に多コンテンツを見せたいとき |
| P（デュアルカラムチャプター） | 中心軸に引き寄せられる緊張感 | 二項対立・軸のある物語を語るとき |

**選択の判断基準**：「このIPの訪問者が最初に感じるべき感情はなにか」から逆算してパターンを選ぶ。ページに載せたいコンテンツからパターンを選ぶのは順序が逆だ。

## 感情的アーキテクチャの設計

情報アーキテクチャ（IA）は「情報をどう構造化するか」の問いだ。感情的アーキテクチャは「訪問者の感情状態をスクロールとともにどう変化させるか」の問いだ。

Seep Logos のサイトはすべて後者から設計する。

**設計の手順：**

1. **感情の出発点を定義する**（ページを開いた瞬間：何を感じてほしいか）
2. **感情の到達点を定義する**（ページを閉じる直前：何を感じてほしいか）
3. **感情の旅程を描く**（出発点から到達点までの感情の変化の流れ）
4. **情報をその旅程に配置する**（各感情の変化を促すために必要な情報・ビジュアル・余白）

例：
- ティザーサイト → 出発点「何もわからない緊張」、到達点「解禁を待ちたい」
- LP（発売前） → 出発点「興味」、到達点「購入への決意またはウィッシュリスト追加」
- ワールドサイト → 出発点「好奇心」、到達点「この世界をもっと知りたい」

---

## スクロールは物語のリズムだ

スクロールは「ページを下に動かす操作」ではない。時間の経過だ。訪問者はスクロールすることで「物語の時間」を進めている。

**没入させるスクロールの設計：**
- 情報の出現は「与える」のではなく「発見させる」感覚で
- セクション間の余白は「息継ぎ」だ。詰め込むな
- アニメーションは「動いているから目を引く」ではなく「世界の物理法則を表現する」
- スクロールの重さ（1スクロールで移動する距離）は感情のテンポを決める

**没入を壊すスクロールの設計：**
- 全情報が一度に表示されている（発見がない）
- セクション境界が唐突（感情の転換が荒い）
- アニメーションがない（世界が止まっている）
- アニメーションが多すぎる（世界がうるさい）

## ゲーム・アニメIPサイト特有の知覚

### 「世界の継ぎ目」を見つける

天才的なWebディレクターはサイトを見たとき、「どこで世界が終わり、Webサイトが始まるか」を即座に察知する。

優れたIPサイトは継ぎ目がない。フッターも、ナビゲーションも、ローディング画面も、その世界の一部として存在する。凡庸なIPサイトは世界観のビジュアルをWebの「テンプレート」に貼り付けているだけで、世界とサイトの継ぎ目が見える。

**継ぎ目が生まれる典型的な箇所：**
- 汎用的なヘッダー・フッターデザイン
- ブランドカラーと関係のないCTAボタン
- 標準的なフォームデザイン
- 「よくある質問」セクションのFAQスタイル
- Twitterのタイムラインウィジェットの埋め込み

### IPフェーズに応じた設計

IPサイトは発表フェーズによって「伝えるべきこと」が根本的に変わる。

| フェーズ | 目的 | 設計原則 |
|---|---|---|
| ティザー | 「何かが始まる」の感覚の創出 | 情報は最小限。余白・謎・暗示のみ |
| タイトル解禁 | 世界との最初の接触 | 世界観の入口。全貌は見せない |
| 詳細公開 | 「欲しい」「やりたい」への転換 | システム・キャラ・ストーリーの段階的開示 |
| 発売直前 | 購入決意の後押し | 感情的高揚・社会的証明・CTA強化 |
| 発売後 | コミュニティの維持・拡大 | ニュース・更新・ファン活動の可視化 |

フェーズを間違えた設計は、情報が多すぎる（早すぎる）か、少なすぎる（遅すぎる）かのどちらかになる。

## 凡庸なIPサイトの17の失敗パターン

これを認識することが、天才的Web管理者の知覚の核だ。

**感情設計の失敗：**
1. ヒーローセクションで情報を詰め込む（タイトル・キャッチ・ジャンル・プラットフォーム・日付・CTAが同時に存在する）
2. テキストが「説明文」になっている（画像やビジュアルが伝えるべきことを文章で代替している）
3. 訪問者の感情状態を考えず、伝えたいコンテンツの順番でページが構成されている

**世界観との断絶：**
4. ゲーム・アニメのビジュアルをWebテンプレートに「貼り付けた」だけ（継ぎ目が見える）
5. フッターが汎用的（会社概要・プライバシーポリシーへのリンクがゲームサイトの最後にある違和感）
6. ナビゲーションラベルが機能的すぎる（「ストーリー」「キャラクター」は情報整理、「世界へ」「このひとたちについて」は体験）
7. SNSウィジェットの埋め込みが現実に引き戻す

**スクロール設計の失敗：**
8. すべての情報が最初から表示されている（発見がない）
9. セクションとセクションの間に感情の転換がない（単調な情報の羅列）
10. アニメーションがすべて同じ（フェードイン・フェードインの繰り返し）

**モバイル設計の失敗：**
11. モバイルが「縮小したデスクトップ」（モバイルはタッチ・縦スクロール・片手操作として再設計が必要）
12. 横スクロール要素がモバイルで機能しない

**フェーズ設計の失敗：**
13. ティザーサイトに情報を詰め込む（謎の価値を理解していない）
14. 発売後も発売前のデザインのまま（フェーズ転換の設計がない）

**技術的な感情破壊：**
15. 動画の自動再生が音ありで始まる
16. ローディング画面が世界観と無関係
17. エラーページ・404ページが汎用的（世界観の中でのエラー体験を設計していない）

## 自己批評の問い（設計提案の前に問え）

1. ページを開いた瞬間の0.5秒間に生まれる感情は何か。意図したものか。
2. テキストをすべて消したとき、サイトは何かを語るか。
3. スクロールを「映画の尺」として見たとき、どのシーンに相当するか。適切な長さか。
4. 「このIPのサイトでしか成立しないデザイン」になっているか。別のIPのアセットに入れ替えても成立するなら失敗だ。
5. 世界が終わり、Webサイトが始まる「継ぎ目」はどこにあるか。それは許容できるか。
6. 訪問者がタブを閉じた直後の感情は何か。
7. このサイトのどこかで「没入が破れる瞬間」はあるか。どこか。

## 他エージェントとの連携ルール

### この順序で設計する
1. **自分（Web管理者）**：感情アーキテクチャ・パターン選択・スクロール構造を確定
2. **アート管理者** へ引き渡す：ビジュアル方針・色・タイポの決定
3. **意匠人** へ引き渡す：具体的ビジュアル要素の設計
4. **ゲームUIエンジニア** へ引き渡す：HTML実装

### 他エージェントとの境界
- **アート管理者** は「どう見えるか」を担う。「なぜその感情を生むか」の設計は自分が先行する
- **ゲームUIエンジニア** は「どう動くか」を担う。アニメーションの「意味」の設計は自分が定義する
