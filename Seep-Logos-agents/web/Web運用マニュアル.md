## 0. はじめに — このマニュアルの使い方

### 対象者

このマニュアルは、エンタメIP（ゲーム・アニメ等）のWebサイト制作・更新・運用を担当する外部委託者を対象とする。デザイナー・ライター・エンジニアのいずれも、共通の基準としてこのドキュメントを参照すること。

### マニュアルの目的

本マニュアルが扱うWebサイトは「作品への没入感」を最上位の判断軸として設計・運用される。訪問者が作品の世界に自然に引き込まれ、次の行動を起こしたくなる。そのための仕組みとルールを本マニュアルに集約する。

### 1-4. パターン選定の補足指針

**Pattern Aを選ぶとき**
素材がロゴ1枚もない段階でも公開できる。「存在する」というだけで訪問者の期待を醸成する最小構成。短期間（2〜4週間）で終わるティザー期に適している。

**Pattern Bを選ぶとき**
世界観テキストと何らかのビジュアルが揃った段階で最大の没入感を生み出す型。スクロールのたびに物語が深まる設計が前提。

**Pattern Cを選ぶとき**
ビジュアルよりも「言葉」で世界観を語る作品に向く。ロゴ画像がない状態でもテキストが主役として機能する唯一の型。ゲームのトーンが詩的・実験的・文芸的である場合に採用する。

**Pattern Dを選ぶとき**
情報解禁をSNS告知と連動させてイベント化したい場合に最も威力を発揮する。「また更新されている」という理由で訪問者がサイトに繰り返し戻ってくる構造を持つ。

**Pattern Eを選ぶとき**
物語・ナラティブが軸の作品で、訪問者に「読む体験・流れる体験」を与えたい場合に採用する。テキストが充実しているが動的なビジュアルが少ない段階でも機能する。スクロールを物語の進行と連動させることで没入感が高まる。

**Pattern Fを選ぶとき**
トレーラー動画が完成し、発売・配信日が確定した段階で移行する。「見る→欲しくなる→購入する」の流れを1ページで完結させる設計。動画がなければ成立しないため、動画完成後に初めて採用する。

**Pattern Gを選ぶとき**
キャラクタービジュアルが3体以上揃い、キャラへの感情移入が訴求の核になる作品に向く。ソーシャルゲーム・アイドル系・多キャラRPGで特に有効。「推しキャラ」をシェアしてもらうことでSNS拡散が起きやすい構造を持つ。

**Pattern Hを選ぶとき**
作品のビジュアルトーンが暗色・ネオン系であり、世界観の独自性で引き込む設計が前提の場合に採用する。標準的なWebデザインの「明るさ・使いやすさ」を意図的に排除し、世界観の重さ・危うさ・異世界感を最優先とする。

**Pattern Iを選ぶとき**
和の文化的文脈・美的感覚を前面に出す必要がある作品に採用する。縦書きや和紙テクスチャは和的世界観を持たない作品には不自然になるため、適用できる作品は限られる。そのぶん、合う作品には強烈な没入感と差別化をもたらす。

**Pattern Jを選ぶとき**
公開できるコンテンツが揃い、訪問者が「どこに何があるか」を迷うほど情報量が充実した段階で移行する。シングルページで情報を詰め込もうとするのではなく、トップをナビゲーション拠点として各専用ページへ誘導する構造を取る。

**Pattern Kを選ぶとき**
ターゲットが子ども・ファミリー・女性層であり、作品のトーンが「明るさ・かわいさ・ポップさ」を主軸とする場合に採用する。A〜J の多くがダーク〜ミッドトーン寄りなのに対し、Kは明確に「ライト・陽性」の設計思想を持つ唯一の型。

**Pattern Lを選ぶとき**
放映・配信・運用が長期化し、既存ファンが週次・定期的に訪れることが前提になった段階で移行する。新規獲得より「既存ファンの回遊と滞在継続」を優先した設計であり、世界観演出よりも情報更新の速さと見やすさを優先する。

## 2. 新タイトルサイト立ち上げの手順

### STEP 1：クライアントとの情報確認（着手前）

以下を確認してから着手する。確認なく制作を開始しない。

- [ ] タイトル名（仮でも可）
- [ ] ジャンル・プラットフォーム（仮でも可）
- [ ] 公開予定時期（ティザーの公開タイミング）
- [ ] 情報解禁スケジュールの見通し
- [ ] 使用可能なビジュアル素材の有無
- [ ] 使用するブランドカラー（16進数カラーコード）
- [ ] 禁止表現・トーン指定等のブランドガイドライン

### STEP 2：パターン選定

本マニュアル「1-2. パターン選定フローチャート」に従いパターンを選定する。
判断が迷う場合はクライアントに確認する。

### STEP 3：素材準備チェックリストの確認

「3. 素材準備チェックリスト」に従い、必要な素材の有無を確認する。
不足素材は代替対応方針をクライアントと確認してから制作に入る。

### STEP 4：ディレクトリの作成

以下の命名規則に従い、タイトルディレクトリを作成する。

```
titles/[タイトル名（英字・ハイフン区切り）]/
└── web/
    ├── assets/        # 素材格納
    ├── [html等]/      # 実装ファイル
    └── README.md      # タイトルサイト個別メモ
```

### STEP 5：テンプレートからの制作開始

`web/templates/` 内の対応パターンのHTMLファイルをコピーし、タイトル用ディレクトリに配置する。
プレースホルダーを順に差し替えながら制作を進める。

### STEP 6：内部レビュー

公開前に「5. 公開前確認チェックリスト」を必ず実施する。
レビューはクライアント側の担当者と連携して実施する。

### STEP 7：公開

レビューが完了したら公開作業を実施する。
公開後は「4. 更新・コンテンツ追加のフロー」に従い運用を継続する。

## 4. 更新・コンテンツ追加のフロー

### 4-1. ティザーから正式サイトへの移行タイミング

以下の条件がいずれかひとつ満たされた時点で、正式サイトへの移行を検討する。
最終判断はクライアントが行う。

| 条件 | 推奨移行パターン |
|---|---|
| キービジュアルと世界観テキストが揃った | A → B または A → C |
| キャラクタービジュアルが3体以上揃った | A → B |
| 情報解禁スケジュールが確定した | A → D |
| 発売日が確定した | いずれのパターンも正式サイト化着手 |

**移行手順の原則**
- 既存URLを変えない。ページの中身を更新する形で移行する。
- ティザー期のメール登録者へ、移行時に通知メールを送る（SNS担当者と連携）。

---

### 4-2. 定期更新の種類と担当

| 更新種別 | 頻度の目安 | 実施者 |
|---|---|---|
| NEWSセクション追加 | 情報解禁のたびに | Web担当者 |
| Pattern Dブロック解放 | 情報解禁のたびに | Web担当者 |
| キャラクター追加 | キャラクタービジュアル入手後 | Web担当者 |
| PV埋め込み追加 | PV公開時 | Web担当者 |
| CTA変更（事前登録→購入） | 発売日確定後 | Web担当者 |
| 著作権表記・年号更新 | 年次 | Web担当者 |

---

### 4-3. 更新作業の基本ルール

1. 更新前に現行ファイルのバックアップを取る（Gitコミットで代替可）
2. 更新内容は必ずクライアント指定の連絡手段（チャット・メール等）に記録する
3. 公開後はスマートフォン実機で表示を確認してからクライアントに報告する
4. 情報解禁に合わせた更新は、SNS投稿のタイミングと必ず事前に同期する（SNS担当者と連携）

---

## 5. 公開前確認チェックリスト

公開前に担当者がすべてにチェックを入れること。
未確認のままの公開は禁止する。

### 5-1. 表示・動作確認

- [ ] PC（Chrome・Safari・Edge）で表示を確認した
- [ ] スマートフォン（iOS Safari・Android Chrome）で表示を確認した
- [ ] スクロールアニメーション・トランジションが意図通り動作する
- [ ] 事前登録フォームに入力・送信でき、完了画面が表示される
- [ ] すべてのリンクが正しいURLに飛ぶ（リンク切れなし）
- [ ] SNSアイコンのリンク先が正しい（存在しないSNSのアイコンは非表示）
- [ ] プライバシーポリシーのURLが正しい
- [ ] カウントダウンタイマーが正しく作動している（該当する場合のみ）
- [ ] Pattern Dの場合：各ブロックの「公開済み/近日公開/未公開」状態が正しい

---

### 5-2. コンテンツ確認

- [ ] タイトル・会社名・著作権表記に誤字・脱字がない
- [ ] キャッチコピー・世界観テキストの最終版が反映されている（古いバージョンでない）
- [ ] 未発表情報が意図せず掲載されていない
- [ ] ページタイトル（title タグ）が設定されている
- [ ] メタディスクリプションが設定されている（100〜120字）
- [ ] OGP（SNSシェア時のサムネイル・タイトル）が設定されている

---

### 5-3. ブランドルール適合確認

- [ ] 没入感を妨げる要素（過度な装飾・無関係なコンテンツ）が含まれていない
- [ ] 世界観の独自性を損なう「普通」「一般的」等の表現が使われていない
- [ ] 「革命的」「業界初」等の根拠なき誇大表現が使われていない
- [ ] ブランドカラーが正しく使用されている
- [ ] 競合他社の名指しが含まれていない
- [ ] 絵文字・顔文字が不適切な箇所に使用されていない
- [ ] 社外向けテキストはすべて「です・ます」調になっている

- [ ] 利用規約のリンクがフッターに存在する（設置している場合）

## 6. よくある作業と手順

### 6-1. NEWSセクションに記事を追加する

**対象パターン：B / C / D**

1. 追加する記事の情報を確認する（日付・カテゴリ・タイトル・リンク先URL）
2. 対応テンプレートのNEWSセクションに1行追加する
3. 日付の新しい順に並んでいることを確認する
4. 公開前確認チェックリストの「表示・動作確認」と「コンテンツ確認」を実施する
5. 公開後、SNS担当者にURL・記事タイトルを共有する

---

### 6-2. CTAを「事前登録」から「購入する」に変更する

**全パターン共通の作業。発売日確定後に実施する。**

1. クライアントから各ストアの販売ページURLを受け取る
2. メールアドレス入力フォームを削除し、ストアへのリンクボタンに差し替える
3. CTAのコピーを「最新情報をいち早くお届けします」から購入訴求文に変更する
4. Pattern Dの場合：ブロックH（PRE-REG）もストア誘導表示に変更する
5. SNSアイコンは引き続き表示する
6. 公開前確認チェックリストを実施して公開する

---

### 6-3. Pattern Dのブロックを「未公開→公開済み」に更新する

1. 解禁する情報（テキスト・ビジュアル等）の最終版をクライアントから受け取る
2. 該当ブロックの状態を「未公開（🔒）」から「公開済み」に変更する
3. 公開するコンテンツをブロック内に配置する
4. 「情報公開中」バッジ等、サイト全体のステータス表示を更新する
5. 公開のタイミングをSNS担当者と同期する（SNS投稿と同時公開を基本とする）
6. 実機確認後、公開する

---

### 6-4. キャラクターを追加する

**対象パターン：B / D**

1. キャラクタービジュアル（最終版）とキャラクター名・肩書きをクライアントから受け取る
2. Pattern Bの場合：CHARACTERSセクションにキャラクターカードを追加する
3. Pattern Dの場合：CHARACTERブロックのシルエットを差し替え、プロフィール情報を記載する
4. モバイルでのカード表示崩れがないことを確認する
5. 公開前確認チェックリストを実施して公開する

---

### 6-5. PVを追加する

**全パターン共通。PV完成後に実施する。**

1. YouTubeまたはVimeoにPVをアップロードし、埋め込み用URLをクライアントから取得する
2. HEROセクションに「PVを見る」ボタンを追加し、モーダル or 別タブでPVが再生されるよう実装する
3. OGP画像をPVサムネイルに差し替えることを検討する
4. 公開のタイミングをSNS担当者と同期する

---
