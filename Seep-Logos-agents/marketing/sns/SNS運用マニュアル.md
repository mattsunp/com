# SNS運用マニュアル

> 作成日：2026-03-26
> 対象：外部委託者（SNS運用担当）

---

## 1. このマニュアルの使い方

### 概要

本マニュアルは、新タイトルのSNSプロモーションを担当する外部委託者が「何から手をつけるか」を迷わずに動けるよう設計した実務ガイドです。

投稿文の作成方法・スケジュールの組み立て方・承認フロー・緊急対応手順をひとつの文書に集約しています。本マニュアルを参照した上で作業することで、投稿品質と世界観の一貫性を担保します。

### 対象者

- 新タイトルのSNS運用を担当する外部委託者
- 投稿文のレビュー・承認を行うクライアント側の責任者

### ファイルマップ

```
marketing/sns/
├── SNS運用マニュアル.md              ← 本ファイル（全体の起点）
├── schedules/
│   ├── announcement-schedule.md     # ティザー〜正式発表当日のスケジュール（ゲーム・アニメ共通）
│   ├── game-posting-schedule.md     # ゲーム：発表後〜リリース後の全フェーズ
│   └── anime-posting-schedule.md   # アニメ：発表後〜放送後の全フェーズ
└── templates/
    ├── game-launch-templates.md     # ゲーム用X投稿テンプレート集
    └── anime-launch-templates.md   # アニメ用X投稿テンプレート集

titles/[作品名]/sns/
└── [作品名]-sns-plan-phase[N].md    # タイトル専用SNSプラン（フェーズ単位で作成）
```

## 2. 新タイトルのSNS運用を始めるときのフロー

新タイトルのSNS運用を開始する際は、以下の手順を順番に実施してください。

### ステップ1：作品情報の確認と整理

まず、以下の情報をクライアント（企画・制作担当者）から取得します。未確定の項目については「未定（後日解禁）」として管理し、確定前に外部へ出ないよう注意します。

| 確認項目 | 備考 |
|---|---|
| 作品タイトル（正式表記） | 表記揺れがないか確認 |
| ジャンル | RPG・アクション / TVアニメ等 |
| プラットフォームまたは放送局 | 発表タイミングの確認も |
| 発売時期または放送時期 | 「未定」の場合も明記 |
| 公式サイトURL | 発表日に稼働するか確認 |
| 公式ハッシュタグ | 他作品と重複しないか検索確認 |
| キービジュアル・ティザー素材 | 解像度・使用許諾の確認 |

### ステップ2：発表タイミングの確定

クライアント側の責任者と「T-0（正式発表日）」を確定します。発表日が決まれば、`announcement-schedule.md` のタイムラインにそのまま当てはめて逆算スケジュールを組むことができます。

### ステップ3：タイトル専用SNSプランの作成

`titles/[作品名]/sns/` ディレクトリを作成し、タイトル専用SNSプランファイルを作成します。ファイル名は `[作品名]-sns-plan-phase1.md` とします。

タイムライン・フェーズ設計・投稿文案をこのファイルにまとめ、クライアントと共有してください。

### ステップ4：各スケジュールファイルをフェーズごとに参照

本マニュアルの「3. スケジュールファイルの使い方」を参照し、作品種別と現在のフェーズに応じたスケジュールファイルを適用します。

### ステップ5：テンプレートから投稿文を起こす

「4. テンプレートファイルの使い方」を参照し、プレースホルダーを実際の情報に差し替えて投稿文を作成します。

### ステップ6：投稿前チェック・承認フローを経て投稿

「5. 投稿前チェックリスト」と「6. 投稿承認フロー」に従って確認・承認を取得した上で投稿します。

## 3. スケジュールファイルの使い方

### 3つのファイルの役割と使い分け

| ファイル | 対象 | カバー範囲 |
|---|---|---|
| `announcement-schedule.md` | ゲーム・アニメ共通 | T-21日（ティザー第1弾）〜T+30日 |
| `game-posting-schedule.md` | 新作ゲーム | 正式発表以降〜リリース後の継続運用 |
| `anime-posting-schedule.md` | 新作アニメ | 制作発表以降〜放送終了後 |

**使い分けの原則は明快です。**

- 発表前のティザー期から正式発表当日まで → `announcement-schedule.md` を使う
- 正式発表が終わった後のゲーム作品 → `game-posting-schedule.md` に引き継ぐ
- 制作発表が終わった後のアニメ作品 → `anime-posting-schedule.md` に引き継ぐ

### ゲームのフェーズ構成

`game-posting-schedule.md` は以下の4フェーズで構成されています。現在の作品が「どのフェーズにあるか」を確認してから開くことで、必要なセクションをすぐに見つけられます。

| フェーズ | 期間目安 | 投稿頻度 |
|---|---|---|
| フェーズ1（情報開拓期） | 発表後〜発売3ヶ月前 | 週2〜3回 |
| フェーズ2（加速期） | 発売3ヶ月前〜1ヶ月前 | 週3〜5回 |
| フェーズ3（クライマックス期） | 発売1ヶ月前〜発売当日 | 週5回〜毎日 |
| フェーズ4（継続運用期） | 発売後〜 | 週3〜5回 |

### アニメのフェーズ構成

`anime-posting-schedule.md` は以下の5フェーズで構成されています。

| フェーズ | 期間目安 | 投稿頻度 |
|---|---|---|
| フェーズ1（世界観開拓期） | 制作発表後〜放送3ヶ月前 | 週2〜3回 |
| フェーズ2（加速期） | 放送3ヶ月前〜1ヶ月前 | 週3〜5回 |
| フェーズ3（クライマックス期） | 放送1ヶ月前〜放送開始 | 週5回〜毎日 |
| フェーズ4（放送中） | 放送期間全体 | 週4〜5回（放送日連動） |
| フェーズ5（余韻醸成期） | 放送終了後 | 週2〜3回 |

### スケジュールを実際の日付に落とし込む手順

1. 「T-0（発表日）」または「発売日」「放送開始日」を確定する
2. スケジュールファイルの各マイルストーンを実際のカレンダー日付に変換する
3. タイトル専用SNSプランファイル（`titles/[作品名]/sns/`）に日付入りの投稿計画として転記する
4. 投稿頻度の推奨曜日（水・木・日等）を参考に、具体的な投稿日を割り当てる

## 4. テンプレートファイルの使い方

### テンプレートの種類

- `game-launch-templates.md`：ゲーム向け（ティザー・発表・リリース・事前登録等）
- `anime-launch-templates.md`：アニメ向け（ティザー・制作発表・放送開始・各話投稿等）

いずれもX（Twitter）専用のテンプレートです。Instagram・TikTok等への展開は各プラットフォームの文法に合わせて別途起こしてください。

### プレースホルダーの差し替え方

各テンプレートには以下の形式でプレースホルダーが記されています。

```
[作品タイトル]    ：ゲームの正式タイトル
[ジャンル]        ：RPG / アクション / ADV 等
[プラットフォーム]：PS5 / Nintendo Switch / PC(Steam) 等
[発売時期]        ：2026年X月発売予定 等
[公式サイトURL]   ：https://...
[ハッシュタグ]    ：#作品タイトル 等
```

差し替えの際は以下の点を確認してください。

- タイトルの正式表記（スペース・大文字小文字・括弧の有無）がクライアントのブランドガイドラインに準拠しているか
- URLが実際に稼働しているか（短縮URL使用の場合はリンク先も確認）
- 発売時期・放送時期の表現が他の告知文書と統一されているか（「2026年春」と「2026年4月」が混在しないよう注意）

### 世界観パートを追加する場合

テンプレートはデフォルトで「告知パート（現実側）」のみで構成されています。世界観パート（架空世界の言葉）を加えたい場合は、本マニュアルの「9. 世界観と告知のセパレート原則」を必ず読んでから加筆してください。セパレートのルールを守らない投稿は承認フローで差し戻します。

## 5. 投稿前チェックリスト

投稿文が完成したら、担当者自身が以下を確認した上で承認者に提出してください。

### 情報の正確性

- [ ] タイトルの正式表記が正しいか（表記揺れ・スペルミスがないか）
- [ ] 発売時期・放送時期の表現が他の告知文書と統一されているか
- [ ] URLが実際に稼働しているか
- [ ] キャスト・スタッフ名の表記が正確か（引用元資料と照合）
- [ ] 未発表情報・ネタバレが含まれていないか

### ブランドルール

- [ ] 「革命的」「業界初」「史上最高」等の根拠なき誇大表現がないか
- [ ] 競合他社・他作品の名指し批判・過度な比較がないか
- [ ] 「普通」「一般的」「よくある」等、世界観の独自性を損なう表現がないか
- [ ] 世界観パートと告知パートのセパレートが守られているか（「9. 世界観と告知のセパレート原則」参照）
- [ ] 区切り線（`──────` 等）を使っていないか（空行による余白が原則）

### 炎上・リスク

- [ ] 誤解を招く表現・皮肉・過度な煽りがないか
- [ ] ユーザーの不満を無視したトーンになっていないか
- [ ] コンシューマー版とスマホ版の情報が混在していないか（ゲームの場合）
- [ ] 場面カットに未放送エピソードのネタバレが含まれていないか（アニメの場合）
- [ ] 著作権・引用ルールに問題がないか

### 文字数・ハッシュタグ

- [ ] X（Twitter）：本文＋ハッシュタグを含めて140字以内に収まっているか
- [ ] ハッシュタグ数がフェーズの推奨数の範囲内か（X：2〜5個、Instagram：最大30個）
- [ ] 公式タグが必ず含まれているか

### ビジュアル（画像・動画がある場合）

- [ ] 画像解像度が推奨サイズを満たしているか（X推奨：1200×675px）
- [ ] ビジュアルの使用許諾が取得済みか
- [ ] テキストが読みやすい配色・フォントサイズになっているか

## 6. 投稿承認フロー

### 基本フロー

```
SNS担当者（投稿文作成・チェックリスト確認）
    ↓
クライアント責任者（内容確認・ブランドルール最終チェック・承認）
    ↓
SNS担当者（承認後に投稿実施）
    ↓
SNS担当者（投稿後に投稿URLをクライアント責任者へ共有）
```

### 各ロールの役割

| ロール | 役割 |
|---|---|
| SNS担当者（外部委託） | 投稿文の作成・チェックリスト確認・投稿実施・投稿URL共有 |
| クライアント責任者 | 内容確認・ブランドルール最終チェック・承認・緊急時の意思決定 |

### 承認のタイムライン目安

- 通常投稿：投稿予定時刻の2時間前までにSNS担当者からクライアント責任者へ提出
- 重大発表（T-0当日・発売当日等）：投稿予定時刻の4時間前までに提出
- 緊急対応（炎上・誤投稿等）：この場合は「8. 炎上・誤投稿時の対応手順」に従う

### 承認の記録

承認はチャット・メール等の書面で残してください。口頭承認のみでの投稿は禁止します。

## 7. 投稿時間帯ガイド

### 優先度と推奨時間帯

| 優先度 | 時間帯 | 理由 |
|---|---|---|
| 最高 | 20:00〜22:00 | ゲーム・アニメ・エンタメ層の活動ピーク。帰宅後リラックスモードで情報感度が高い |
| 高 | 18:00〜20:00 | 帰宅途中〜夕食時。スマホ閲覧が増加する時間帯 |
| 中 | 12:00〜13:00 | 昼休み。短時間で情報を取る層が多い |
| 補助 | 8:00〜9:00 | 発売当日・重大発表当日の「宣言投稿」のみ使用 |
| 避ける | 1:00〜7:00 | 深夜〜早朝。リーチが著しく低下するため原則使用しない |

### 重大発表日（T-0・発売当日・放送開始日）の特例

重大発表日は例外的に1日3〜4投稿が許容されます。ゴールデンタイム（20:00）に本発表を置き、前後を補足投稿で挟む構成が標準です。

```
8:00〜9:00   「本日発表します」宣言
20:00        【最重要】タイトル・ビジュアル・基本情報の全公開
21:00〜22:00  公式サイト誘導
22:00〜23:00  反響感謝・続報予告（任意）
```

### 同日複数投稿の間隔ルール

- 同日に複数投稿する場合は最低4時間の間隔を空ける
- 1日5投稿を超えるとフォロワーのミュート・フォロー解除リスクが高まる
- 発表当日・発売当日は上記の例外として3〜4投稿を許容する

### プラットフォーム別の補足

- X（Twitter）：20:00〜22:00 を最優先。拡散速度が最も高い時間帯
- Instagram：20:00〜22:00 に加え、土日の12:00〜13:00 も有効
- スマホアプリゲームのフェーズ4（継続運用期）：12:00 と 20:00〜22:00 の2段構えが有効

## 8. 炎上・誤投稿時の対応手順

### 誤投稿（誤字・誤情報・未発表情報の流出等）を発見した場合

**第一報：発見から5分以内**

1. SNS担当者は即座にクライアント責任者へ状況を報告する（内容・投稿時刻・現在の反響規模）
2. クライアント責任者の指示が出るまで追加投稿・コメント返信は停止する

**削除の判断**

3. クライアント責任者が削除の是非を判断する
4. 誤情報・未発表情報の流出・著作権違反が明確な場合は速やかに削除する
5. 誤字程度の軽微なミスの場合は、削除ではなく訂正投稿で対応することも選択肢とする

**訂正投稿の文例**

```
先ほどの投稿に誤りがありました。正しくは以下の通りです。

[正しい情報]

ご確認いただいたみなさまにお詫び申し上げます。
```

**事後報告**

6. 対応完了後、SNS担当者はクライアント責任者へ対応結果の報告を行う
7. 再発防止のため、チェックリストの見直しを行う

---

### 炎上（批判の集中・不適切表現の指摘等）が発生した場合

**第一報：確認から10分以内**

1. SNS担当者は問題の投稿と批判内容を保存した上でクライアント責任者へ即時報告する
2. 批判のトーンと規模（リツイート数・返信数・メディア取材の有無）を把握する
3. クライアント責任者の指示が出るまで、その投稿へのコメント返信・引用投稿は行わない

**初動対応の原則**

- 拡散中の炎上に対してSNS上でその場の言い訳・反論はしない
- 事実確認が終わる前に謝罪投稿を出すことも避ける（誤情報が広がる可能性がある）
- 深刻な問題（差別・プライバシー侵害・法的リスク）を含む場合はクライアント側の顧問弁護士・上位責任者に即時エスカレーションする

**削除の判断**

- 批判内容が正当で、表現上の問題が明確な場合は削除して謝罪投稿へ移行する
- 誤解に基づく批判の場合は、削除せず事実確認の説明投稿を検討する

**謝罪・説明投稿の基本構成**

```
[問題となった内容]についてご不快をおかけしたことをお詫び申し上げます。

[事実関係の説明（1〜2文）]

今後はより一層注意を払って参ります。
```

**対応記録の保存**

- 対応した全投稿・削除した内容・対応時系列をクライアント指定の記録先に保存する

## 9. 世界観と告知のセパレート原則

### 原則

エンタメIP（ゲーム・アニメ等）のSNSプロモーションでは、1投稿内に**世界観パート（架空世界の言葉）**と**告知パート（現実の情報）**が共存することがあります。ただし、両者が文章の流れの中で溶け合った状態になることを禁じます。境界は空行による余白で作ります。

### セパレートのルール

**区切り線は使わない。空行で間を作る。**

線は主張しすぎます。空白はただそこにあります。受け手は余白の中で自然に世界観から現実へと切り替わります。

### NG例

```
傘は、雨粒から守るためだけにあるわけじゃない。続報をお待ちください。
```

世界観の言葉と告知の言葉が一続きになっています。読み手はどこまでが作品の言葉で、どこからが告知なのかを判断できません。

### OK例

```
傘は、雨粒から守るためだけにあるわけじゃない。

新作ゲーム『[作品タイトル]』近日公開。続報は公式アカウントでお知らせします。
```

空行1行が境界を作っています。世界観パートはそこで完結し、告知パートはそこから始まります。

### 適用の判断基準

| 投稿要素 | 役割 | 文体の目安 |
|---|---|---|
| 世界観パート | 世界・キャラクター・物語の断片を届ける | 詩的・余白のある表現。設定の全量は開示しない |
| 告知パート | 情報解禁・サイト誘導・リリース告知等 | 端的・情報量重視。「です・ます」調または体言止め |

### 世界観パートを書く際の注意

- 世界観パートは「世界の断片」を届けるものです。設定を全量説明しようとしないでください
- フォロワーの想像力を刺激する余白を意図的に残します
- キャラクターの言葉・世界の空気感・物語の予感を短い言葉で切り取ることを意識します
- 世界観パートが長すぎると告知パートが霞みます。1〜3文を目安にします

### 告知パートを書く際の注意

- 告知パートは読者が迷わず行動できる情報を含めます（URL・時刻・ハッシュタグ等）
- 世界観パートの余韻を壊さないよう、過剰な感嘆符・煽り表現は使いません
- 「詳細は公式サイトへ」「続報は公式アカウントでお知らせします」等の定型文は有効です
