**各記事の主な知見：**

- 「AI部下10人」記事：将軍・家老・足軽8名の階層マルチエージェントシステム「multi-agent-shogun」。Claude Code × tmux、イベント駆動、YAML通信。Claude Max ×5（月額$100）での運用実績。

- 「家老が切腹しかけた話」記事：コンテキスト圧縮時にルールが抜け落ちる問題が発生。3層コンテキスト管理（Memory MCP・グローバル・プロジェクト）で対処。Extended Thinking無効化、スキルは白紙から育てる設計。

- 「CLAUDE.mdが500行になった理由」記事：7つの重大事故（Cドライブ削除未遂、フレンドリーファイア、役割誤認等）のたびにルールを追記した経緯を記録。核心的な指摘として「ルールを書いて外部記憶に頼う時点で、思考そのものは改善されない」「LLMに何をさせるか設計する人間側の力がボトルネック」が示された。

- 「コンテキストアーキテクチャ」記事：マルチエージェントの本質問題はオーケストレーションではなくコンテキスト管理であると定義。Lost in the Middle問題、N²通信問題を提示。判断基準のLoRA化、最小Human-in-the-Loopが解法として示された。

- 「ハーネスエンジニアリング」記事：ほとんどの実装はGit Workflowのbash焼き直し（層1〜2）であると分類。未踏領域は層3：マルチエージェント協調・安全装置設計。

- 「CoDD」記事：仕様変更時の整合性維持をMarkdownフロントマターの依存宣言と依存グラフで解く手法を提示。`pip install codd-dev`、`codd impact <ファイル>` で波及範囲を自動特定できる。

**考察まとめ：**

- 6記事は「経験→理論→解決」の一本の論考として構成されている。
- 現Seep Logosリポジトリも同様の問題を内包している。すなわち、CLAUDE.md と agents/*.md の依存関係が暗黙的であり、変更時の波及を人力で追う構造になっている。
- 「ルールを積む」方向ではなく「依存構造を設計する」方向が正しいアプローチであるとの認識が共有された。
