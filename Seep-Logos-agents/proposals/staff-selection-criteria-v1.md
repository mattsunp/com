## 2. 選抜の背景と方針

### 背景

Seep Logos は自スタジオの制作力を外部向けサービスとして展開するにあたり、サービス品質の核となる2つの職域スタッフを既存メンバーから選出する。外部採用ではなく社内アサインとするのは、スタジオの世界観とブランドトーンをすでに内在しているメンバーを優先するためである。

対象サービスは以下のとおり。

- Webサイト制作サービス（テンプレート準拠型。内製コーディング対応含む）
- SNS運用サービス（X / Instagram / Discord の3プラットフォーム対応）

### 選抜方針

> 「スキルより、何を作るために生きているかを先に問う」

以下の3原則に基づいてアサイン候補者を評価する。

1. **世界観への共鳴**：Seep Logos のIP・ブランドメッセージを外部に向けて体現できるか
2. **自律的な業務遂行**：フルコミットではない中で、限られた工数を自己管理しながら品質を維持できるか
3. **クライアント対応耐性**：外部クライアントとのやり取りにおいて、スタジオの代表として誠実に振る舞えるか

## 3. Webデザイナー枠 選抜基準

### 3-1. 必須スキル（Must）

- HTML / CSSの読み書きができる（新規フレームワーク設計は不要。テンプレート準拠の微細な修正が対象）
- Figma または同等のデザインツールでビジュアルデザインデータを作成・修正できる
- SP / PC両対応のデザイン調整（レスポンシブ対応）について基本的な理解がある
- クライアントへのデザイン案提示・フィードバック反映の経験がある（社内外問わず）
- 納期・修正ラウンドの管理を自主的に行える

### 3-2. 歓迎スキル（Want）

- AI補助ツール（Cursor / GitHub Copilot 等）を活用した制作経験がある
- JavaScript の基礎的な読み書きができる
- エンタメ・IPコンテンツ関連のWebデザイン制作経験がある
- テンプレート設計・コンポーネント整理の経験がある
- クライアントヒアリングから要件整理までを一貫して担った経験がある

### 3-3. NG要件

- 「デザインは感性で作るもの」と考え、ロジックや言語化を避ける傾向がある
- 修正指示に対して感情的になりやすく、クライアントとの調整を苦手とする
- 工数・締切への意識が薄く、作業ステータスの共有が自発的にできない
- スタジオのブランドトーンや世界観に対して無関心、または軽視する姿勢がある

### 3-4. 人物要件

- 「このクライアントのIPをどう見せるか」という視点でデザインに向き合える
- 自分のデザインの意図を言葉で説明できる（クライアント説明・社内レビュー共通）
- スタジオの一員として、外部クライアントにSeep Logosの水準を示す責任感を持てる
- 完成よりも「届けること」を優先できる現実的なバランス感覚がある

### 3-5. 想定レンタル工数

| フェーズ | 期間目安 | 月次工数 | 週次換算 | 備考 |
|---|---|---|---|---|
| 初期フェーズ | サービス開始〜3ヶ月 | 70h / 月 | 約17.5h / 週 | 初月のみ初期投資工数（テンプレート整備等）17h含む |
| 定常フェーズ | 4ヶ月目以降 | 55h / 月 | 約13.5h / 週 | 月2案件標準スループット想定 |

> フルコミットではないため、既存業務との工数バランスをアサイン前に個別確認すること。

## 4. SNS運用テキスト制作者枠 選抜基準

### 4-1. 必須スキル（Must）

- X（旧Twitter）/ Instagram の投稿文・キャプション制作の実務経験がある
- ブランドのトーン＆ボイスを理解した上で、プラットフォームごとに文体を使い分けられる
- 投稿スケジュールを自律的に管理し、期日どおりに制作物を提出できる
- ビジュアル（画像・ビジュアルデータ）を伴う投稿の制作または補助ができる
- 月次レポート（インプレッション・エンゲージメント等の定量情報）をまとめられる

### 4-2. 歓迎スキル（Want）

- Discord サーバーの構築・運用経験がある
- コミュニティ設計（チャンネル構成・イベント企画・メンバー育成）の経験がある
- エンタメ・ゲーム・アニメ等のIPコンテンツ関連アカウントの運用経験がある
- AI補助（文章生成・要約・翻訳等）を活用した制作効率化の経験がある
- SNS分析ツール（Sprout Social、Buffer、Metricool 等）の使用経験がある

### 4-3. NG要件

- 投稿文を「作業」として消化し、世界観や文脈への配慮なく量産する傾向がある
- コミュニティ内のトラブル・炎上兆候に対して適切に対応できない、またはエスカレーションが遅い
- プラットフォームの特性（X と Instagram と Discord では求められるコミュニケーションが根本的に異なる）を理解せず画一的に対応しようとする
- スタジオや担当IPへの興味・愛着がなく、業務としてのみ取り組もうとする

### 4-4. Discord コミュニティ運用に関する追加要件

Discordは「投稿型」ではなく「コミュニティ運用型」のプラットフォームである。他2プラットフォームとは業務の性質が本質的に異なる。以下の要件を別途確認すること。

**追加必須要件**
- Discordのサーバー構造（チャンネル・ロール・権限設定）を理解している
- モデレーション業務（スパム対応・ルール遵守の維持・問題ユーザーへの対応）を自律的に行える
- コミュニティの「温度感」を定性的に把握し、運営方針に反映できる観察眼がある

**追加歓迎要件**
- Discordボット（MEE6、Carl-bot 等）の設定・管理経験がある
- AMAセッション・クイズ・交流イベント等のコミュニティ施策を企画・運営した経験がある
- ファンコミュニティまたはIPコンテンツのDiscordサーバーに参加・運営した経験がある

**留意点**
- Discordは規模拡大に伴いモデレーション工数が急増するリスクがある（月20hを超過した場合は体制見直しを検討）
- 炎上・スパム多発時の対応ガイドラインは初期投資フェーズ内にSNS制作者が整備する

### 4-5. 人物要件

- 「このIPのファンが何を求めているか」を考えながら言葉を選べる
- コミュニティに対してオーナーシップを持ち、「自分のコミュニティを育てる」感覚で向き合える
- 定性情報（コミュニティの雰囲気・ファンの反応）を言語化してレポートできる
- 3プラットフォームを並走させながら品質を落とさない段取り力がある

### 4-6. 想定レンタル工数

| フェーズ | 期間目安 | 月次工数 | 週次換算 | 備考 |
|---|---|---|---|---|
| 初期フェーズ | サービス開始〜2ヶ月 | 90h / 月 | 約22.5h / 週 | 初月に初期投資工数（テンプレート整備・Discord設計等）20h含む |
| 定常フェーズ | 3ヶ月目以降 | 70h / 月 | 約17.5h / 週 | X: 20h / Instagram: 26h / Discord: 19h（バッファ込み） |

> フルコミットではないため、既存業務との工数バランスをアサイン前に個別確認すること。Discordの規模拡大時は追加工数が生じる可能性がある点を候補者に事前に説明すること。

## 5. 共通評価軸

両職域に共通して用いる4つの評価軸。各軸を5段階で評価し、総合スコアと面談での定性判断を組み合わせてアサインを決定する。

### 軸 1：世界観適合性

| 評価視点 | 確認方法 |
|---|---|
| Seep Logos のミッション・ブランドメッセージを自分の言葉で説明できるか | 面談での問い（「Seep Logos が作るものに、あなたはどんな価値を感じていますか」） |
| 担当するIPの世界観に対して自発的な関心・愛着があるか | 過去の発言・制作物・リアクション等から推測 |
| クライアントへの対応においてSeep Logosの看板を背負う意識があるか | 面談での確認 |

> 「仕事だから対応できます」ではなく、「このIPの世界観を届けたい」という内発的な動機があるかを見極めること。

### 軸 2：自走力

| 評価視点 | 確認方法 |
|---|---|
| 指示がなくても作業の優先順位を自分で判断して動けるか | 過去業務での実績・エピソードヒアリング |
| 問題が生じた際に自分でまず解決策を考えてから相談できるか | 面談での問い |
| 作業ステータスを自発的に共有する習慣があるか | 過去の業務でのコミュニケーション傾向を確認 |

### 軸 3：外部クライアント対応耐性

| 評価視点 | 確認方法 |
|---|---|
| 修正依頼・フィードバックに対して感情的にならず建設的に対応できるか | 面談での問い・過去の対外業務経験ヒアリング |
| クライアントの要望を整理し、優先度を付けて対応できるか | 面談での問い |
| 「スタジオの代表」としての振る舞いを意識できるか | 面談での言動・姿勢から総合判断 |

### 軸 4：工数管理の自律性

| 評価視点 | 確認方法 |
|---|---|
| 週17〜22h（初期フェーズ）の工数を、既存業務と並走しながら自律的に管理できるか | 現在の業務量・スケジュール確認 |
| 工数が超過しそうな場合に早期にアラートを出せるか | 面談での問い |
| タスクの見積もりと実績の乖離を振り返り、改善できる傾向があるか | 過去実績・傾向ヒアリング |

## 6. 選抜フロー

```
Step 1: 候補者リストアップ
  └ 担当マネージャーまたはプロデューサーが既存スタッフの中から
    各枠の必須スキルを充足する候補者を1〜3名リストアップする

Step 2: 書面確認（ポートフォリオ・実績確認）
  └ 候補者の過去制作物・業務実績を確認し、
    必須スキルの実質的な充足度を評価する
  └ NG要件に抵触する傾向がないかをここで一次スクリーニングする

Step 3: 面談（30〜45分）
  └ 4つの共通評価軸に基づいた問いかけを実施する
  └ 想定工数・業務内容・期待値を候補者に明示し、合意を確認する
  └ Discord対応が必要な場合は、SNS枠候補者に対して
    Discord関連の追加要件を個別に確認する

Step 4: 評価・合議
  └ 面談実施者が共通評価軸（4軸、各5段階）でスコアを記録する
  └ 総合スコアと定性コメントをプロデューサーに提出する
  └ プロデューサーが最終アサインを決定する

Step 5: 候補者への説明・合意形成
  └ アサインされる候補者に対して、業務内容・工数目安・期待値を
    文書で明示した上で口頭でも説明し、双方向の合意を形成する
  └ 既存業務との優先度ルールを明確にしておく

Step 6: 初期フェーズ開始・モニタリング
  └ アサイン後2ヶ月間は週次で工数・業務進捗を確認する
  └ 問題が生じた場合は早期に体制を見直す
```

## 7. 工数上限アラートライン

クライアント数の増加に伴い、以下のラインを超えた場合は増員または体制変更の検討を開始する。

### Webデザイナー

| 状況 | アラートライン | 対応 |
|---|---|---|
| 月次案件数が3件を超える | 月80h超過 | project-manager にエスカレーション。増員または案件受注数の上限設定を検討 |
| 修正ラウンドが平均3回超（月次集計） | 月90h超過リスク | クライアントへの仕様確認プロセス見直しと、仕様変更の追加工数ルール整備 |
| 案件並走数が常時2件を超える | 品質低下リスク | 案件受注タイミングの調整。同時並走2件を上限とするルール設定 |

### SNS制作者

| 状況 | アラートライン | 対応 |
|---|---|---|
| Discordモデレーション工数が月20hを超過する | 定常フェーズ月90h超過 | Discord対応の専任化または補助スタッフのアサインを検討 |
| クライアント数が2社以上になる | 月120h超過リスク | 1人での3プラットフォーム×複数クライアント対応は限界を超える。増員前提で受注計画を立て直す |
| 投稿品質のばらつきが顕在化する | 定性アラート | プラットフォーム別優先順位（X > Instagram > Discord の緊急度）を再確認し、作業集中ルールを設ける |

### 共通：増員判断基準

以下の条件が1つでも継続して2ヶ月以上発生した場合、プロデューサーは採用・クリエイター運用部と協議し増員判断を行う。

- 想定月次工数の120%超過が継続している
- 担当スタッフが「既存業務を圧迫している」と報告している
- 納品品質のクライアントからのフィードバックスコアが低下している
